# Audit MVP - Hallu Block - 2026-07-02

Audit realise sur le depot local `/home/cladstriff/projects/hallu-block`.

## Verdict

Le MVP est bien structure et le coeur fonctionnel est plutot sain pour une extension MV3 : permissions limitees au besoin, pas de telemetry applicative, packaging allowlist propre, moteur teste sur fixtures, et canary live deja en place.

Je ne recommanderais pas une sortie store sans traiter les points P0 ci-dessous. Les risques principaux ne sont pas le moteur de masquage lui-meme, mais la readiness release : discours privacy trop absolu par rapport au fetch distant, validation insuffisante des regles hot-fix, lint non vert, dependances de build vulns via `web-ext`, et divergence entre les regles bundlees et les regles publiees.

## P0 - A traiter avant sortie

### 1. Le message privacy "aucune donnee ne sort" est trop absolu

References :

- `src/background/service-worker.js:29`
- `src/background/service-worker.js:60`
- `src/background/service-worker.js:62`
- `_locales/fr/messages.json:60`
- `_locales/en/messages.json:60`
- `README.md:5`

Le service worker fetch quotidiennement `https://rozieres.github.io/hallu-block/rules.json` pour les hot-fixes. C'est probablement acceptable produit si c'est clairement documente, car l'URL est statique et ne contient pas l'historique de navigation. Mais les textes actuels disent :

- "100 % local, zero tracking"
- "Aucune donnee ne sort de votre navigateur."
- "No data ever leaves your browser."

Ce n'est pas strictement vrai : une requete reseau sort du navigateur vers GitHub Pages. Meme si aucune donnee personnelle applicative n'est envoyee volontairement, l'appel expose a minima IP, user-agent, timing et installation active.

Impact :

- Risque de rejet ou friction Chrome Web Store si les champs privacy, la fiche store et le comportement divergent.
- Risque de perte de confiance utilisateur si quelqu'un inspecte les permissions et voit `https://rozieres.github.io/*`.

Recommandation :

- Remplacer le wording par une formulation exacte, par exemple : "Aucune page consultee, aucun contenu de navigation et aucun compteur ne sont envoyes. L'extension telecharge periodiquement un fichier public de regles de detection depuis GitHub Pages pour corriger les selecteurs."
- Ajouter une page `docs/privacy.md` ou une page publique equivalente, liee depuis la fiche store.
- Envisager un toggle "regles automatiques" si vous voulez conserver une promesse strictement locale.

### 2. Les regles distantes sont trop peu validees

References :

- `src/background/service-worker.js:60`
- `src/background/service-worker.js:64`
- `src/background/service-worker.js:65`
- `src/background/service-worker.js:70`
- `src/background/service-worker.js:71`
- `src/content/engine.js:263`
- `src/content/engine.js:281`

`refreshRemoteRules()` accepte n'importe quel JSON distant si :

- `version` est une string,
- `hide` est un array,
- la version est strictement superieure.

Le moteur applique ensuite les `selectors`, `ancestorSelectors`, `hosts`, `path`, `textHeadings` et `annotMode` fournis. Ce n'est pas de l'execution de JavaScript distant, mais c'est quand meme une configuration qui controle ce que l'extension masque sur Google, Bing, DuckDuckGo, YouTube et Amazon.

Scenario de risque :

- Compromission du compte GitHub, de GitHub Pages, ou erreur de publication.
- Publication d'une regle `selectors: ["body"]` sur un host autorise.
- L'extension cache une page entiere ou des parties non liees a l'IA chez tous les utilisateurs ayant recu la config.

Impact :

- Defacement/censure involontaire de pages.
- Risque review MV3 : les politiques Chrome demandent que la logique complete soit discernable depuis le code soumis ; les ressources externes peuvent etre des donnees, mais elles ne doivent pas devenir un interpreteur de commandes distantes.

Recommandation :

- Ajouter un schema strict local avant adoption :
  - IDs autorises uniquement : `google-ai-overview`, `google-ai-mode`, `bing-copilot`, `youtube-ask`, `amazon-rufus`.
  - Hosts autorises par ID, et sous-ensemble des hosts bundlees.
  - Selecteurs interdits : `html`, `body`, `*`, selecteurs trop generiques sans borne.
  - `annotMode` enum : absent ou `css`.
  - `annotateKey` dans une allowlist locale.
  - `slopFilter` valide ou ignore.
- Versionner et publier les regles avec un test de schema.
- Option plus robuste : signer le fichier distant, ou supprimer le hot-fix distant pour v0.1 si la promesse privacy prime.

### 3. `docs/rules.json` diverge deja de `src/rules/rules.json`

References :

- `docs/rules.json:45`
- `src/rules/rules.json:47`
- `src/rules/rules.json:48`
- `tests/popup.spec.js:254`

Diff observe :

```diff
-      "annotateKey": "annot_generic"
+      "annotateKey": "annot_generic",
+      "annotMode": "css"
```

`docs/rules.json` semble etre la source publiee par GitHub Pages pour `REMOTE_RULES_URL`. Elle n'a pas `annotMode: "css"` pour Bing, alors que le bundle l'a. Comme les versions sont egales (`2026.07.01`), les clients n'adoptent pas aujourd'hui la copie distante. Mais le premier bump de `docs/rules.json` sans resynchronisation ferait perdre le mode CSS removal-proof pour Bing.

Le test existant verifie seulement que la version servie n'est pas derriere le bundle. Il ne detecte pas les divergences fonctionnelles a version egale.

Recommandation :

- Faire de `src/rules/rules.json` la seule source de verite.
- Generer `docs/rules.json` depuis `src/rules/rules.json` dans le script de release.
- Ajouter un test de deep equality quand `version` est identique, ou interdire deux fichiers manuels.

### 4. `npm run lint` n'est pas une gate verte

References :

- `package.json:13`
- `scripts/pack.sh:35`
- `scripts/pack.sh:36`
- `manifest.json:30`
- `manifest.json:31`

`npm run lint` echoue avec :

```text
MANIFEST_FIELD_UNSUPPORTED "/background/service_worker" is not supported.
```

Le commentaire dans `scripts/pack.sh` indique que c'est attendu pour Firefox/`web-ext`, mais le script retourne quand meme exit 1. En l'etat, `lint` ne peut pas etre une commande de validation pre-release.

Impact :

- Signal CI/release brouille : une commande nommee lint echoue par design.
- La mention "Firefox-ready" est fragile tant que le validateur Firefox rejette le service worker MV3.

Recommandation :

- Si v0.1 est Chrome-first : renommer en `lint:firefox-known-failing` ou remplacer par une validation Chrome-targeted.
- Si Firefox est dans le scope release : produire un manifest Firefox compatible ou une variante build dediee.
- Ajouter une commande `release:check` qui doit etre strictement verte.

### 5. Audit npm complet : vulnerabilites critiques dans la toolchain de dev

References :

- `package.json:20`
- `package.json:22`
- `package-lock.json` via `web-ext`

`npm audit --omit=dev` est vert, mais `npm audit` complet remonte 8 vulnerabilites, toutes via `web-ext` et ses dependances :

- `shell-quote` critique via `fx-runner`
- `tmp` high
- `ajv` moderate
- `uuid` moderate via `node-notifier`

`npm audit` propose `web-ext@10.4.0`, breaking change.

Impact :

- Pas d'impact runtime direct dans l'extension publiee, car ces paquets ne sont pas dans le ZIP.
- Impact supply-chain et CI : la toolchain qui construit/lint l'artefact a des dependances vulnerables.

Recommandation :

- Tester l'upgrade `web-ext` vers 10.x sur une branche.
- Si upgrade impossible avant v0.1, documenter explicitement que le risque est build-only, et isoler les jobs CI sans secrets inutiles.
- Ajouter `npm audit --omit=dev` et `npm audit` dans la checklist release avec politique claire.

### 6. Accessibilite popup insuffisante pour les toggles

References :

- `src/popup/popup.html:24`
- `src/popup/popup.html:25`
- `src/popup/popup.html:39`
- `src/popup/popup.html:60`
- `src/popup/popup.js:146`
- `src/popup/popup.js:150`
- `src/popup/popup.js:153`

La plupart des interrupteurs sont des `<li>` cliquables. Ils n'ont pas de role, pas de `tabindex`, pas de `aria-pressed`/`aria-checked`, et ne sont pas actionnables au clavier. Seul le bouton radical `udm14` a un vrai `<button>` et `aria-pressed`.

Impact :

- Utilisation difficile ou impossible au clavier/screen reader.
- Les etats `[█]` / `[ ]` sont visuels seulement.

Recommandation :

- Remplacer chaque ligne toggle par un `<button type="button">` ou un vrai `<input type="checkbox">`.
- Ajouter `aria-pressed` ou `aria-checked`, `aria-describedby` pour le sous-texte.
- Tester navigation clavier : Tab, Space, Enter.
- Ajouter quelques assertions Playwright sur les roles ARIA.

### 8. Couverture Google geographique limitee et incoherente avec DNR

References :

- `manifest.json:18`
- `manifest.json:22`
- `manifest.json:49`
- `manifest.json:53`
- `src/rules/dnr/udm14.json:16`
- `src/rules/rules.json:6`
- `src/rules/rules.json:24`

Les content scripts et host permissions couvrent `google.com`, `google.fr`, `google.be`, `google.ch`, `google.ca`. Le ruleset DNR `udm14` matche en revanche `google\.[a-z.]{2,}`. En pratique, sans host permission et content script, la promesse "Google" ne s'applique pas a beaucoup de TLDs (`google.co.uk`, `google.de`, etc.).

Impact :

- Fonctionnalite percue comme aleatoire selon pays.
- Permission/review plus difficile a raisonner car le DNR exprime plus large que le manifest.

Recommandation :

- Soit assumer et documenter "Google FR/BE/CH/CA/COM".
- Soit etendre proprement les match patterns et tests aux TLDs cibles.
- Aligner regex DNR et hosts autorises.

### 9. Pas de `minimum_chrome_version`

References :

- `manifest.json:1`
- `manifest.json:92`
- `src/content/baseline-google.css:18`
- `src/rules/rules.json:27`

Le manifest fixe `strict_min_version` pour Firefox (`128.0`) mais aucun `minimum_chrome_version`. Le code depend de fonctionnalites modernes : MV3 service worker, DNR `queryTransform`, `:has()` en CSS/selecteurs, etc.

Impact :

- Des utilisateurs sur versions Chrome anciennes peuvent installer une extension partiellement cassee.

Recommandation :

### 12. Risque performance sur sites tres dynamiques

References :

- `manifest.json:72`
- `manifest.json:78`
- `src/content/engine.js:334`
- `src/content/engine.js:335`
- `src/content/engine.js:336`
- `src/content/engine.js:338`

Le moteur observe `document.documentElement` avec `subtree: true` et relance des `querySelectorAll` sur chaque burst de mutations. Sur YouTube et Amazon, les pages sont tres dynamiques. Les content scripts matchent toutes les pages YouTube et Amazon, pas seulement les pages ou les widgets cibles sont attendus.

Impact :

- Cout CPU potentiel sur longues sessions YouTube/Amazon.
- Risque de degradation percue meme quand aucun bloc IA n'existe.

Recommandation :

- Restreindre les matches si possible : YouTube `/watch*`, Amazon `/dp/*`, `/gp/product/*`, etc.
- Ajouter une limite : stop observer apres X secondes sans detection sur certains hosts, ou observer seulement des conteneurs connus.
- Ajouter un micro-benchmark Playwright sur une fixture avec mutations frequentes.

### 16. Le parser anti-slop accepte des tokens qui ne sont pas des hosts

References :

- `src/background/service-worker.js:99`
- `src/background/service-worker.js:104`
- `src/rules/slop/noai_hosts.txt:2139`
- `src/rules/slop/noai_hosts.txt:4289`
- `tests/engine.spec.js:406`

Exemples dans la liste :

```text
0.0.0.0 youtube.com/@LaCarnevali
0.0.0.0 www.youtube.com/@LaCarnevali
```

Le parser garde tout token qui contient un point. Ces entrees contiennent un chemin et ne matcheront jamais un `hostname` dans `isBlocked()`. Ce n'est pas dangereux, mais c'est du bruit et cela rend la taille/qualite de la liste moins claire.

Recommandation :

- Nettoyer avec `new URL("https://" + token).hostname` ou une regex host stricte.
- Dedupliquer les domaines apres normalisation.
- Etendre le test pour refuser `/`, `:`, espaces et wildcard non supporte.

## Points solides

- Permissions raisonnables pour le scope actuel : `storage`, `declarativeNetRequest`, `alarms`, pas de `tabs`, pas de `<all_urls>`.
- Pas de `innerHTML` ou `eval` dans le code extension. Le popup et les annotations utilisent `textContent`.
- Packaging allowlist robuste dans `scripts/pack.sh`; le ZIP inspecte ne contient pas de fichiers internes.
- Le moteur fail-open si le service worker ou la blocklist ne repondent pas.
- Compteur local, stocke dans `storage.local`, avec serialisation des writes.
- Tests fixtures riches : Google AI Overview, Google AI Mode, anti-slop Google/DDG, Bing Copilot, YouTube, Amazon, popup, DNR shape.
- Canary live present pour detecter drift structurel, meme s'il faut ameliorer la semantique des skips.
